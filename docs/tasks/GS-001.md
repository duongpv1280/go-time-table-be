## Overview

A user can sign in or sign up to the system using their Google Account. The frontend handles the Google OAuth flow and sends the resulting Google ID token to the backend via a REST API. The backend verifies the token, resolves the user's identity and role, and returns the user's profile together with their permissions.

**Type**: Feature

## Context

The frontend obtains a Google ID token through the Google OAuth 2.0 flow (e.g. Google Sign-In button). It then calls `POST /api/v1/auth/google` with that token. The backend must:

1. Verify the Google ID token against Google's public keys.
2. Extract the user's email and name from the token claims.
3. Look up the user by email in the database.
   - If found → sign in: return existing profile and permissions.
   - If not found → sign up: create a new user record, assign a default role, then return profile and permissions.
4. Return a structured response on success, or a typed error on failure.

The first admin account is seeded at startup from environment config (see `docs/specs/rbac-explanation.md §7`). Role assignment for new users arriving via Google OAuth is `STUDENT` by default; the admin can promote users afterwards.

## Target | Goal

`POST /api/v1/auth/google` — verify a Google ID token and return the authenticated user's profile and scopes.

### Request

```json
{
    "idToken": "eyJhbGciOiJSUzI1NiIsImtpZCI6..."
}
```

### Success Response — `200 OK`

```json
{
    "name": "Mr. A",
    "email": "teacher.a@school.com",
    "role": "TEACHER",
    "permissions": {
        "classes": {
            "students": "read"
        },
        "teachers": "read",
        "slots": "write"
    }
}
```

`role` is one of `ADMIN`, `TEACHER`, `STUDENT`. `permissions` is derived from the role via the Casbin policy table at auth time (not fetched from a separate endpoint).

### Error Response — `401 Unauthorized`

```json
{
    "error": "invalid_token",
    "message": "Google ID token is invalid or expired"
}
```

### Error Response — `500 Internal Server Error`

```json
{
    "error": "internal_error",
    "message": "An unexpected error occurred"
}
```

## Out of Scope

- Frontend OAuth flow (button, redirect, consent screen) — handled entirely by FE.
- Issuing a session JWT from the backend — this task only returns user info + permissions; session token design is a separate task.
- Role promotion / user management endpoints — covered by a separate admin task.
- Refresh token handling.
- Social login providers other than Google.

## Acceptance Criteria | Definition of Done

- [ ] `POST /api/v1/auth/google` is defined in `api/paths/auth.yaml` and `api.gen.go` is regenerated without errors (`make oapi-codegen`).
- [ ] Handler (`handlers/auth.go`) binds the request body, delegates to the usecase, and maps domain errors to the correct HTTP status codes (`401` for token errors, `500` for unexpected failures).
- [ ] `IGoogleVerifier` interface is defined in the domain or usecase layer; the real implementation in `infrastructure/auth/google_verifier.go` calls Google's token verification endpoint.
- [ ] `IGoogleAuthUseCase.Execute(ctx, idToken)` verifies the token, upserts the user (create on first login with role `STUDENT`), and returns `AuthResponse` with `name`, `email`, `role`, and `permissions`.
- [ ] `permissions` in the response is derived from the user's Casbin role at auth time and reflects the policy table (not hardcoded).
- [ ] New user created via sign-up appears in the `users` table with a `casbin_rule` `g` row assigning them the `STUDENT` role.
- [ ] Invalid or expired Google ID token returns `401` with `error: "invalid_token"`.
- [ ] **Unit tests** (`usecase/auth/usecase_test.go`): `IGoogleVerifier` and `IUserRepository` are mocked; covers sign-in path, sign-up path, invalid token path, and DB error path.
- [ ] **Integration tests** (`infrastructure/auth/google_verifier_test.go`): hits the real Google tokeninfo endpoint with a known-invalid token and asserts the correct error is returned; uses real SQLite DB for repository-layer tests.
- [ ] All new types, interfaces, and file names follow `CLAUDE.md` conventions (PascalCase types prefixed with `I` for interfaces, kebab-case file names, camelCase variables).
- [ ] `make test` passes with no failures.
- [ ] `make build` compiles cleanly after `make generate`.
