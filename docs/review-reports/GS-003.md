## Executive Summary

The GS-003 implementation introduces a clean, reusable validator infrastructure (`IValidator`/`rules.Validator`) on top of `go-playground/validator/v10`, with two custom rules (`unique_in` and `notblank`) and two new protected HTTP endpoints (`POST /api/v1/classes`, `PUT /api/v1/classes/:classId`). All four Acceptance Criteria are satisfied and all 109 tests pass. The only notable findings are a test-plan/implementation disagreement on HTTP status codes for role-rejected requests (401 vs 403), uncovered error branches in `UpdateClass` and `CreateClass`, and a silently discarded error from validator registration — none of which are blockers or criticals.

## 5. Security Review

**SQL injection via `unique_in` table/column names:**

The `uniqueInValidator` at `unique_in.go:19-23` builds a GORM query as:
```go
q := v.db.WithContext(ctx).Table(table).Where(column+" = ?", fl.Field().String())
```
`table` and `column` originate from `fl.Param()`, which returns the text after `=` in the struct tag (e.g., `unique_in=classes:name` → `fl.Param()="classes:name"`). Struct tags are compiled into the binary at build time, not derived from runtime user input. The actual user-supplied field value (`fl.Field().String()`) is correctly passed as a parameterized argument (`?`), so there is no injection path from external callers.

The risk surface exists only at developer time: if a developer writes a struct tag pointing to a table or column name derived from user input (which is structurally impossible with Go struct tags), or if a malicious developer introduces a tag like `unique_in='; DROP TABLE classes; --:name`. Since struct tags require source-code access, this is a supply-chain risk, not a runtime vulnerability. The pattern is as safe as it can be within the go-playground/validator extension model.

**Authentication and authorisation:**
- All new POST and PUT routes are registered on the `apiV1` group which has `httpMiddleware.JWTAuth` applied (`main.go:34`). No unprotected registration path was found.
- `CreateClass` and `UpdateClass` use cases enforce `permission.Role == RoleAdmin` before any repository call (`usecase.go:106`, `usecase.go:126`). Non-ADMIN roles receive `ErrUnauthorized` and the handler maps this to HTTP 401.
- The anti-enumeration pattern (mapping `ErrClassNotFound` to `ErrUnauthorized` in `UpdateClass`) is correctly implemented at `usecase.go:135-136`.

**Sensitive data / error leakage:**
- The `validationError` function (`class.go:172`) returns `fe.Tag()` (the tag name, e.g. `"unique_in"`) as the field message — this leaks which rule failed per field. This is intentional and expected for 422 responses; it does not expose internal implementation details beyond what the API consumer needs.
- 500 responses return the generic string `"An unexpected error occurred"` with no internal error message — correct.

## 7. Warnings

**Findings:**

1. [WARNING] `registerCustomValidations` at `validator.go:26-29` silently discards the errors returned by `RegisterValidationCtx` and `RegisterValidation`. Both functions in `go-playground/validator/v10` return `error`. A registration failure (e.g., due to a name conflict or invalid tag name) would cause the rule to silently not register, and all subsequent validations using that tag would pass without executing the custom logic — a dangerous silent failure. The errors should be checked and either panicked on (acceptable in an init path) or returned to the caller.

2. [WARNING] `(*classUseCase).GetClassByID` has cognitive complexity 18 (limit: 12). See Section 4. Pre-existing, but GS-003 provides a natural opportunity to note it for a future task.

3. [WARNING] Test plan (`GS-003-test-plan.md`) specifies HTTP 403 with `error: "forbidden"` for TEACHER and STUDENT role rejections on POST/PUT (TC-064, TC-065). The implementation returns HTTP 401 with `error: "unauthorized"`. The actual integration tests (`integration_test.go:405`, `:421`) were written to assert 401, so all 109 tests pass. However, the API contract documented in the test plan differs from the deployed behaviour. The correct status for "authenticated but insufficient role" is debatable (403 is the strict RFC interpretation; 401 is the anti-enumeration stance the codebase uses uniformly). The implementation is internally consistent but the test plan should be updated to reflect the 401 convention, or the convention itself should be re-evaluated as a separate task.

4. [WARNING] `unique_in=classes:name` in `types.go` hardcodes the table name `classes`. If the GORM model's `TableName()` ever changes (e.g., to `class_records`), the validator tag will silently query the wrong table. This is an acceptable coupling for now but should be documented in the struct or in a named constant (e.g., `const ClassesTable = "classes"`).

## 8. Bug Heuristics

**Findings:**

- [INFO] `gormClassRepository.Update` uses GORM `Updates()` with a struct, which skips zero-value fields. `Grade` is an `int`, and domain enforcement (`NewGrade` requires g > 0) guarantees the value is never zero. `Name` is a string — `NewName` requires a non-empty trimmed string. This is safe given the domain invariants, but would silently fail to update if those invariants were relaxed without updating the repository method.

- [INFO] In `UpdateClass` usecase (`usecase.go:140`), when `name == ""` the name update is skipped entirely. Combined with the `UpdateClassRequest` struct having `validate:"omitempty,unique_in=classes:name"` for `Name`, a PUT request with an omitted `name` field will skip validation and skip the domain update — the grade alone is updated. This is the intended partial-update behaviour and is tested by `TestUpdateClass_OnlyGrade_UpdatesGrade`. No bug, but the semantics differ from a POST where name is required.
