## 5. Security Review

**Findings:**

- [INFO] **Route security confirmed fixed**: Class routes are exclusively on the JWT-protected `apiV1` group. `RegisterHandlers` does not register `/api/v1/classes` or `/api/v1/classes/:classId`. There is no unprotected registration of class routes. The prior `[CRITICAL]` from Re-run 1 is resolved.

- [INFO] All SQL queries in `class-repository.go` use GORM parameterised queries (`?` placeholders at lines 27, 47, 56, 69, 78, 87). No injection risk.

- [INFO] JWT Verify uses algorithm check (`if _, ok := t.Method.(*jwt.SigningMethodHMAC); !ok` at `jwt-service.go:44`), preventing algorithm-confusion attacks where a forged RS256 token is accepted by an HS256 verifier.

- [INFO] No hardcoded secrets. `JWT_SECRET` is read from environment via `config.Load()`.

- [INFO] 5xx responses return `"An unexpected error occurred"` — no internal details leaked.

- [INFO] No sensitive fields (passwords, tokens used internally) appear in log output. `AuthResponse.Token` is intentionally returned to the caller.

- [INFO] **Nil claims defensive gap** (noted in Re-run 1 as an optional improvement, not a confirmed crash): `JWTAuth` at `middleware/auth.go:33` checks `claims.UserID == "" || claims.Role == ""` without a prior nil check on `claims`. - [INFO] No goroutine leaks. No goroutines started without cancellation paths in the changed code.

## 7. Warnings

**Findings:**

- [WARNING] `GetClassByID` for TEACHER role fetches all classes for the teacher via `FindByTeacherUserID` and filters by ID in memory (`usecase.go:75-85`). For teachers with a large number of class assignments this is O(n) in memory. A `FindByIDForTeacher(ctx, classID, teacherUserID)` repository method would be more efficient. Not a bug; deferred to future work.

- [WARNING] `ClassModel.Name` carries a `uniqueIndex` constraint (`class-model.go:11`). The domain layer does not enforce this constraint, so a duplicate-name insert will return a raw GORM/SQLite error rather than a typed domain error. Not a GS-002 defect, but the inconsistency between domain and DB constraints is worth addressing in a future task.

- [INFO] `TestJWTAuth_NonBearerHeader_Returns401` uses `"Basic abc123"` (a header with a non-Bearer scheme). TC-005 in the test plan specifies a raw token without any prefix scheme. Both inputs produce identical behaviour at the `strings.HasPrefix(..., "Bearer ")` check, so the test is logically correct. The additional case (`"sometoken.without.prefix"`) is covered at 100% middleware coverage because the same branch is hit. No action required.

- [INFO] TC-008 (unknown role passes middleware) is implicitly covered by `TestJWTAuth_ValidToken_CallsNextAndSetsPermission` (which injects a valid TEACHER claim and confirms `next` is called) in combination with the usecase tests rejecting unknown roles. No dedicated middleware test by that name exists, but the branch (middleware passes any valid-format token) is covered within the 100% package coverage.
